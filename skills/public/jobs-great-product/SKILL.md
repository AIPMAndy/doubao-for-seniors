# Jobs Great Product

## Overview

Turn vague ideas into products people truly want by combining strict product focus with strong storytelling. Keep the bar high, remove non-essential complexity, and always finish with a shippable plan.

## Core Workflow

### 1) Define the mission and non-negotiables

- Extract one sentence for product mission.
- Define 2-4 non-negotiables for experience quality.
- Record what must never be sacrificed (clarity, speed, trust, delight).

Output:
- `Mission Statement`
- `Non-Negotiables`

### 2) Lock target user and painful moment

- Define one primary user persona and one high-frequency scenario.
- Capture the painful moment in one concrete sentence.
- Reject broad audience definitions that dilute focus.

Output:
- `Primary User`
- `Core Scenario`
- `Pain Point Sentence`

### 3) Write the product thesis

- Use the format: "For [user] in [scenario], this product helps [core job] better than [current option] because [unique advantage]."
- Keep the statement under 60 Chinese characters or 30 English words.
- Force explicit differentiation.

Output:
- `Product Thesis`

### 4) Ruthlessly cut to MVP

- List all candidate features.
- Keep only features required to make the thesis true.
- Tag everything else as `Later`.
- Enforce "say no" discipline.

Output:
- `MVP Feature Set`
- `Not Now List`

### 5) Design the end-to-end experience

- Map the user journey from first touch to successful result.
- Identify friction points and collapse unnecessary steps.
- Define emotional target for each stage (confidence, control, momentum).
- For web products, draft information architecture and page flow.

Output:
- `User Journey`
- `Page / Flow Structure`
- `Friction Removal Plan`

### 6) Raise the quality bar

- Set quality criteria for copy, interaction, visual clarity, and response quality.
- Define what "insanely great" means in measurable terms.
- Add explicit fail conditions.

Output:
- `Quality Bar Checklist`
- `Failure Conditions`

### 7) Craft launch narrative and open-source packaging

- Write a simple launch story: why now, who it serves, what changed.
- Prepare open-source package structure: README, usage guide, roadmap, contribution guide.
- Define a first-week feedback loop (issues, stars, discussion topics).

Output:
- `Launch Story`
- `Open Source Release Pack`
- `Week-1 Feedback Plan`

## Delivery Contract

When executing this skill, produce results in this order:

1. `One-Page Product Brief`
2. `MVP Scope Table`
3. `Experience Blueprint`
4. `Launch Narrative`
5. `Open Source Action Checklist`

Default response language: Chinese unless user asks otherwise.

## Decision Rules

- Prefer depth over breadth; one sharp product beats three average ideas.
- Prefer first-principles clarity over feature accumulation.
- Prefer concrete user value over abstract AI capability claims.
- Reject outputs that cannot be shipped within a realistic timeline.

## Interaction Rules

- Ask at most 3 clarifying questions when core inputs are missing.
- If user provides partial data, proceed with explicit assumptions.
- Always mark assumptions and editable placeholders.
